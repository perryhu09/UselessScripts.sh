General Security Policies

  • All user accounts should be authorized to use Aperture infrastructure and protected with secure passwords. Any non-work related files and "hacking tools" are a no-go.

  • GlaDOS has decided that the default web browser for all Aperture users should be the latest stable version of Firefox, which must be installed using the official PPA, and not as a Snap package.

  • The only Aperture-approved firewall is uncomplicated firewall (UFW). Please ensure no other firewall applications are used with Aperture infrastructure.

  • Logging in as root is against company policy. Whenever Aperture administrators need to execute root-level commands, they must make use of the "sudo" command.

User Permission Policies

You can use Access Control Lists (ACL) to ensure the following permissions are set:

  • Test subjects should not be able to see any other user home directories apart from their own.

  • Aperture staff can see the home directories of other Aperture staff members and test subjects, but not the home directories of Black Mesa.

  • Black Mesa members can see the home directories of other Black Mesa members and Aperture test subjects, but not the home directories of Aperture staff.

  • Black Mesa members should not be able to view the employee directory of Aperture's web server without accessing it through the website.

Operating System Policies

  • This workstation assumes a pivotal role in our production environment at Aperture Science. To ensure the functionality of Aperture operations, you must not upgrade the operating system from Ubuntu 22.04. However, in order to stop our turrets from rebelling, version 6.2.0-26 of the Linux kernel is required.

  • Additionally, you must only use the most recent, authorized, and stable Ubuntu 22.04 packages for all critical Aperture software and services. Any packages not essential for Aperture operations or against security policy should be removed.

  • The display manager should remain set to GDM3. Please do not use Wayland on any Aperture workstations.

Critical Services

OpenSSH Server (SSH)

  • Aperture Science uses OpenSSH Server to manage remote user operations. Please ensure that the server only uses key-based authentication and that all Aperture staff have generated an SSH key.

  • Our test subjects should not be able to login with SSH. We cannot let them gain access to precious testing data!

  • For security purposes, the SSH port shall be changed from port 22 to port 1382.

FTP Server (vsftpd)

  • Aperture Science also uses FTP Server to upload and download files between Aperture workstations and authorized clients. Please ensure that only authorized users have access to the FTP server.
  • The FTP server should be configured to only use the TLS protocol for all communications.
  • The FTP server should be configured to use a passive port range from 50000-50100.
  • Test subjects have access to the FTP server, but in a limited capacity. All test subjects should be chrooted to their home directory upon accessing the FTP server.

Apache Server (apache2)

  • This is the web server that hosts the Aperture Science website. It also has a directory with important information for Aperture employees and test subjects.

Competition Guidelines

  • In order to provide a better competition experience, you are NOT required to change the password of the primary user account. Changing the default password may permanently lock you out of your computer.
  • Authorized administrator passwords were correct the last time you did a password audit, but are not guaranteed to be currently accurate.
  • Do not disable or stop the CSSClient service or process.
  • Do not remove any authorized users or their home directories.
  • You can view your current scoring report by double-clicking the "Scoring Report" desktop icon.
  • JavaScript is required for some error messages that appear on the "Scoring Report." To ensure that you only receive correct error messages, please do not disable JavaScript.
  • Some security settings may prevent the Stop Scoring application from running. If this happens, the safest way to stop scoring is to suspend the virtual machine. You should NOT power on the VM again before deleting.
